# REQ-004 真机全流程测试用例
*更新：2026-03-25 | 状态：🟢draft | Owner：@test_writer / Sartre*

## 1. 目标与适用范围
- 目标：提供一份可直接执行的 REQ-004 真机手工回归脚本，验证“退出/掉线重连（30秒窗口）”在最小闭环范围内可稳定复现、可观察、可留证。
- 适用需求：`REQ-004 退出/掉线重连（30秒窗口）`。
- 适用范围：
  - `单Worker / 单房间 / Android真机 + service测试环境`
  - 覆盖 `session.reconnect`、`session.reconnected`、`room.recover_hint`、`/sessions/{session_id}/recover`
  - 覆盖窗口内恢复、超窗自动重入房、原麦位不可恢复/需补偿提示、后台回前台恢复、礼物与榜单最终态校验
- 不适用范围：
  - 跨 Worker / 多实例自动恢复
  - 真实 RTC 媒体链路 100 并发重连压测
  - iOS、多机型分层、系统来电抢占、长后台 `>30秒` 场景

## 4. 连接与建链检查

### 4.2 reconnect token 获取检查
- 前置条件：
  - 已完成登录与入房
- 步骤：

- 预期结果：

  3. 执行一次送礼，确保出现礼物订单最终态与榜单变化。
- 预期结果：

  - 至少一条礼物订单进入最终态
  - 榜单发生可见更新

## 5. 完整业务流程用例
以下用例默认基于同一条主链路执行：
`登录 -> 入房 -> 获取 reconnect token -> 上麦或收听 -> 完成一次送礼 -> 触发异常 -> 观察恢复或重入房`

### 5.1 TC-004-01 窗口内断网 5 秒恢复
- 前置条件：
  - 账号 A 已入房
  - 已成功获取 reconnect token
  - 已完成上麦或收听
  - 已完成一次送礼
- 步骤：
  1. 开启录屏。
  2. 关闭真机网络或开启飞行模式，保持约 `5秒`。
  3. 恢复网络。
  4. 观察 App 状态、事件日志与房间页面。
- 预期结果：
  - UI 进入 `RECONNECTING` 后自动恢复
  - 出现 `session.reconnect`
  - 服务端返回 `session.reconnected resume_ok=true`
  - 用户回到原房间，不需要手动重新入房
  - 麦位、订阅状态、最近礼物结果与榜单保持一致
- 关键日志/截图证据：
  - 真机录屏
  - `session.reconnect -> <session_id>`
  - `session.reconnected -> resume=true`
  - 恢复后房间截图、榜单截图
  - 服务端日志中的恢复请求与成功结果

### 5.2 TC-004-02 窗口内断网 25 秒恢复
- 前置条件：同 TC-004-01
- 步骤：
  1. 在房间稳定状态下断网 `25秒`。
  2. 恢复网络。
  3. 观察是否仍在原恢复窗口内成功恢复。
- 预期结果：
  - 不出现卡死、空白页或需要手工重走主流程
  - `session.reconnected resume_ok=true`
  - 若需要补偿，允许先出现 `room.recover_hint`，随后通过快照补齐房态
  - 最近一次礼物订单最终态与榜单快照一致
- 关键日志/截图证据：
  - 录屏
  - `session.reconnected`
  - 若触发补偿：`room.recover_hint`
  - 恢复后礼物订单/榜单截图

### 5.3 TC-004-03 超过窗口 35 秒自动重入房
- 前置条件：同 TC-004-01
- 步骤：
  1. 断网 `35秒`。
  2. 恢复网络。
  3. 观察客户端是否自动进入重入房流程。
- 预期结果：
  - 不应伪装为恢复成功
  - 应出现超窗语义：`RECON_003` 或 `rejoin_required=true`
  - 客户端自动执行重新入房
  - 重入房后仍可继续操作，且最近礼物最终态、榜单快照可见
- 关键日志/截图证据：
  - 录屏
  - `session.reconnected requires rejoin` 或 `RECON_003`
  - 自动重新入房后的房间截图
  - 服务端恢复失败与重入房日志

### 5.4 TC-004-04 原麦位不可恢复 / 需补偿提示
- 前置条件：
  - 账号 A 已上麦
  - 辅助账号 B 可协助占位或触发麦位变化
- 步骤：
  1. 让账号 A 上麦并保持房间在线。
  2. 触发账号 A 断线进入恢复窗口。
  3. 在恢复前通过辅助操作让原麦位不可恢复，或制造需要补偿/重订阅的场景。
  4. 恢复账号 A 网络。
- 预期结果：
  - 不出现“看起来恢复成功但实际上不能说话”的假恢复
  - 若原麦位已失效，应收到明确提示：
    - `RECON_005`
    - 或 `room.recover_hint`
    - 或 `need_resubscribe / need_snapshot_pull`
  - 客户端回到可执行状态：听众态、可重试申请麦位、或可继续完成补偿
- 关键日志/截图证据：
  - `RECON_005`
  - `room.recover_hint`
  - `session.reconnected` 中的 `need_resubscribe` / `need_snapshot_pull`
  - 恢复后麦位状态截图

### 5.5 TC-004-05 后台 25 秒后回前台恢复
- 前置条件：同 TC-004-01
- 步骤：
  1. 保持房间稳定并开始录屏。
  2. 将 App 切后台，保持约 `25秒`。
  3. 回到前台。
  4. 观察是否自动触发恢复。
- 预期结果：
  - 前台恢复后无需手工重走登录/入房
  - 能看到 `session.reconnect`
  - 恢复后房间仍可继续听/说/送礼
  - 榜单与最近礼物最终态一致
- 关键日志/截图证据：
  - 前后台切换录屏
  - `session.reconnect`
  - `session.reconnected`
  - 恢复后操作截图

### 5.6 TC-004-06 礼物与榜单最终态校验
- 前置条件：
  - 任一恢复成功或超窗重入房用例执行完成
  - 用例过程中至少发生 1 次送礼
- 步骤：
  1. 记录断线前最后一次送礼订单与榜单状态。
  2. 完成恢复或自动重入房。
  3. 校验恢复后礼物结果与榜单是否一致。
- 预期结果：
  - 不出现“已扣费但界面无结果”
  - 最近礼物订单保持最终态
  - 榜单快照与服务端一致
  - 若通过恢复接口补偿，应能观察到补偿后状态归一
- 关键日志/截图证据：
  - 断线前礼物与榜单截图
  - 恢复后礼物与榜单截图
  - `gift.accepted` / `gift.broadcast` / `leaderboard.updated`
  - 若存在：`room.recover_hint` 与恢复后的快照结果

## 9. 判定原则
- `PASS`：
  - 能按预期进入恢复或自动重入房
  - 无幽灵会话、双占位、长期静音黑洞
  - 礼物与榜单最终态一致
- `FAIL`：
  - 超窗后未进入明确降级路径
  - 原麦位不可恢复但客户端无提示
  - 恢复后不可听/不可说/不可继续送礼
  - 出现礼物扣费结果缺失或榜单错乱
- `BLOCKED`：
  - 服务端未启动成功
  - 真机包无法安装
  - 测试账号、网络或房间资源未准备齐全
