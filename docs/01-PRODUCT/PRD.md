## 1. 项目概述
本项目是面向实时陪伴社交场景的`商用语聊房MVP`，目标是以`Android原生 + Node.js SFU`快速验证以下两条能力：
- `业务闭环`：用户完成登录/开户、入房、充值、送礼、收入确认。
- `体验闭环`：用户在多人语聊房内获得稳定低延迟语音体验，并在断网/切后台后可恢复会话。

### 1.1 业务目标
- 搭建可执行的礼物收入主链路，形成首充与首礼转化。
- 建立房间社交基础秩序，让用户可持续停留、上麦与互动。
- 通过可追溯账务与基础风控，为后续商业化扩展预留空间。

### 1.2 技术目标
- `100并发`房间下端到端语音延迟`P95 < 300ms`。
- 语聊房支持`8麦位`常态互动，听众可稳定收听。
- 在`30秒恢复窗口`内尽可能恢复房间会话状态。
- 服务SLA目标`99%`。

## 2. 版本边界

- `REQ-001` 注册/登录 + 钱包开通。
- `REQ-002` 创建/加入房间 + 礼物消费闭环。

- 空间音频、AI变声、AI降噪等高级特效。

## 4. 竞品结论与产品策略
详细分析见`/Users/yuanye/myWork/ChatRoom/docs/01-PRODUCT/COMPETITIVE_ANALYSIS.md`。

### 4.1 核心判断
- 本项目应对标“`语聊房社交 + 礼物变现`”品类，而非做大而全的泛娱乐平台。
- MVP阶段最重要的不是功能丰富，而是让主链路足够短、稳定、可追踪。
- 对目标场景而言，`稳定音频体验`和`充值/送礼可靠性`比复杂活动玩法更直接影响转化。

### 4.2 产品策略
- 优先级排序：`登录安全` > `送礼闭环` > `音频稳定` > `重连恢复` > `活动/订阅扩展`。
- 交互策略：降低首次送礼阻力，强化余额不足时的充值引导与首礼反馈。
- 技术策略：所有体验类需求必须绑定量化指标，不接受“感觉可用”的验收方式。
- 运营策略：先依赖推荐房、基础榜单、节日礼物，不引入重型活动系统。

- 首充承接：余额不足时可拉起充值页，支持首充礼包展示。
- 合规提示：虚拟币及礼物仅限站内使用，不支持现金等价流通。

**验收标准**
- 登录成功率（7日滚动）`>=99%`。
- 首登开户成功率`>=99.9%`。
- token签名校验通过率`=100%`；篡改/过期Token拒绝率`=100%`。
- refresh token仅哈希存储，旧会话轮转失效时效`<=60s`。
- `gift.send -> gift.accepted -> gift.broadcast`链路已可联调。

### 6.3 REQ-003 群聊音频（8麦位，100并发）
**目标**
- 在房间多听少说的典型模式下，保证可听、可控、低延迟。
- 让房主和上麦用户在普通移动网络环境中可持续互动。

**当前流转**
- 当前状态：`🟢待设计`。
- 当前负责人：`@arch_perf_dev`。

**核心流程**

2. 上麦用户发布音频流，听众订阅音频流。
3. 服务端按活跃说话人与优先级下发订阅策略。

- 弱网处理：优先保活音频，低优先级订阅可降级到4路。

**验收标准**
- `100并发（8麦 + 92听众）`下语音延迟`P95 < 300ms`。
- 上麦成功率`>=97%`。
- 5分钟内语音连续可听率`>=98%`。

### 6.4 REQ-004 退出/掉线重连（30秒窗口）
**目标**
- 减少网络抖动、App前后台切换、短时断网带来的房间流失。
- 在恢复链路中保持关键状态一致，避免用户重复操作。

**当前流转**
- 当前状态：`🟢待设计`。
- 当前负责人：`@arch_perf_dev`。
- 核心依赖文档：`docs/00-ENTRANCE/DEMAND_ENTRANCE.md`、本节需求说明。
- 当前备注：依赖 REQ-003 会话模型；产品侧已补齐“本周最小闭环”口径，架构/开发阶段需按该范围冻结会话快照、重连 Token、事件补偿与超窗降级路径。

**核心流程**
1. 客户端识别信令断连或媒体断流，进入`RECONNECTING`。
2. 客户端携带`reconnect_token + last_seq`请求恢复。
3. 服务端校验恢复窗口和会话快照，决定恢复或要求重新入房。
4. 恢复成功后执行重订阅与关键事件回补。
5. 超过窗口则重新加入房间，并补偿关键状态。

**功能需求**
- 入房签发`reconnect_token`，默认TTL `30秒`。
- 记录会话快照：麦位、订阅列表、最近事件序号。
- 状态机：`CONNECTED -> RECONNECTING -> RECOVERED / REJOIN_REQUIRED`。
- 补偿机制：礼物、榜单、麦位等关键事件按序回放。
- 提示机制：用户可见的恢复提示与失败降级提示。

**本周最小可执行闭环（P0）**
- 范围先收敛为`单Worker / 单房间 / Android真机 + service测试环境`，目标是把“真机可恢复”闭环跑通，而不是一次性解决多实例切换。
- 真机主路径固定为：`登录 -> 入房 -> 上麦或收听 -> 完成一次送礼 -> 断网/切后台/切网 -> 恢复原会话或超窗自动重入房`。
- Android侧本周必须持有并恢复：`room_id`、`session_id`、`reconnect_token`、`last_seq`、`seat_intent`、最近一次`producer/consumer`上下文。
- 服务端本周必须做到：断线后保留`30秒`恢复窗口，不得在窗口内立即清空会话快照；窗口内优先恢复原会话，超窗后返回明确降级信号并允许自动重新入房。
- 本周恢复补偿的关键状态最少覆盖：`麦位状态`、`订阅状态`、`最近礼物订单最终态`、`当前榜单快照`；其余扩展事件允许后续补证。

**本周不纳入P0**
- 跨 Worker / 多实例切换后的自动恢复不纳入本周通过条件；若未实现，文档与测试结论必须显式标注。
- 真实RTC媒体链路的100并发重连压测，继续依赖`REQ-003`后续补证，不作为本周“单房真机恢复闭环”前置阻塞。
- 大规模机型分层、系统来电抢占、极端长后台（`>30秒`）以外的恢复优化，留在后续批次收敛。

**关键状态一致性口径**
- `麦位一致性`：恢复后同一麦位不得出现双占位；若原麦位已失效，客户端必须收到明确提示并进入可重试状态。
- `订阅一致性`：恢复成功后客户端应回到可听状态；若服务端要求重订阅，必须给出可执行补偿指令，而不是停留静音黑洞。
- `礼物/榜单一致性`：恢复后客户端看到的最近礼物订单最终态、贡献榜快照需与服务端一致，不允许“已扣费但界面无结果”。
- `会话一致性`：超过`30秒`恢复窗口后，必须明确进入`REJOIN_REQUIRED`，禁止表现为“仍在线但不可操作”的幽灵会话。
