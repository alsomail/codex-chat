## 1. 项目概述
本项目是面向实时陪伴社交场景的`商用语聊房MVP`，目标是以`Android原生 + Node.js SFU`快速验证以下两条能力：
- `业务闭环`：用户完成登录/开户、入房、充值、送礼、收入确认。
- `体验闭环`：用户在多人语聊房内获得稳定低延迟语音体验，并在断网/切后台后可恢复会话。

### 1.1 业务目标
- 搭建可执行的礼物收入主链路，形成首充与首礼转化。
- 建立房间社交基础秩序，让用户可持续停留、上麦与互动。
- 通过可追溯账务与基础风控，为后续商业化扩展预留空间。

### 1.2 技术目标
- `100并发`房间下端到端语音延迟`P95 < 300ms`。
- 语聊房支持`8麦位`常态互动，听众可稳定收听。
- 在`30秒恢复窗口`内尽可能恢复房间会话状态。
- 服务SLA目标`99%`。

## 2. 版本边界

### 2.1 In Scope（MVP必须完成）
- `REQ-001` 注册/登录 + 钱包开通。
- `REQ-002` 创建/加入房间 + 礼物消费闭环。
- `REQ-003` 群聊音频（8麦位，100并发）。

- 视频直播、多人视频通话、混流录播。
- 复杂公会体系、跨房PK、任务中心、重活动运营平台。
- 礼物提现到法币、复杂现金分成。
- 空间音频、AI变声、AI降噪等高级特效。

## 4. 竞品结论与产品策略
详细分析见`/Users/yuanye/myWork/ChatRoom/docs/01-PRODUCT/COMPETITIVE_ANALYSIS.md`。

### 4.1 核心判断
- 本项目应对标“`语聊房社交 + 礼物变现`”品类，而非做大而全的泛娱乐平台。
- MVP阶段最重要的不是功能丰富，而是让主链路足够短、稳定、可追踪。
- 对目标场景而言，`稳定音频体验`和`充值/送礼可靠性`比复杂活动玩法更直接影响转化。

### 4.2 产品策略
- 优先级排序：`登录安全` > `送礼闭环` > `音频稳定` > `重连恢复` > `活动/订阅扩展`。
- 交互策略：降低首次送礼阻力，强化余额不足时的充值引导与首礼反馈。
- 技术策略：所有体验类需求必须绑定量化指标，不接受“感觉可用”的验收方式。
- 运营策略：先依赖推荐房、基础榜单、节日礼物，不引入重型活动系统。

## 6. 详细需求说明

### 6.1 REQ-001 注册/登录 + 钱包开通
**目标**
- 新用户在一次会话内完成“登录 -> 开户 -> 入房 -> 准备送礼”的最短路径。
- 建立安全、可追溯的身份与资产基础。

**现状与阻塞**
- 当前状态为`🟡draft`。
- 已知阻塞：`OTP回退口令`、`Token明文存储`、`接口路径偏差`。
- 发布原则：此类问题属于P0门禁，修复前不得将REQ-001判定为通过。

**核心流程**
1. 用户选择手机号OTP登录（MVP主路径）。
2. 服务端完成身份校验并签发JWT/refresh token。
3. 首次登录自动完成钱包开户与风险等级初始化。
4. 用户进入推荐房列表并具备查看余额、拉起充值的基础能力。

**功能需求**
- 登录方式：手机号OTP；第三方登录能力预留，不作为首版必交付项。
- 会话安全：设备指纹、登录频控、Token轮转、异地登录提示。
- 钱包能力：余额、冻结金额、累计消费、近30天消费概览。
- 首充承接：余额不足时可拉起充值页，支持首充礼包展示。
- 合规提示：虚拟币及礼物仅限站内使用，不支持现金等价流通。

**验收标准**
- 登录成功率（7日滚动）`>=99%`。
- 首登开户成功率`>=99.9%`。
- token签名校验通过率`=100%`；篡改/过期Token拒绝率`=100%`。
- refresh token仅哈希存储，旧会话轮转失效时效`<=60s`。
- `gift.send -> gift.accepted -> gift.broadcast`链路已可联调。

### 6.2 REQ-002 创建/加入房间 + 礼物消费闭环
**目标**
- 用户在房间内稳定完成“选礼物 -> 扣币 -> 广播 -> 榜单更新”。
- 平台可对每笔礼物交易进行追踪、回滚与对账。

**核心流程**
1. 房主创建房间并配置公开/私密、主题、标签。
2. 用户通过推荐或搜索进入房间，建立信令与音频会话。
3. 用户打开礼物面板，余额不足时拉起充值。
4. 服务端执行余额校验、幂等校验、风控校验并原子扣币。
5. 礼物动效、房间消息、贡献榜同步更新。
6. 订单进入可追踪状态，用于对账与异常回溯。

**功能需求**
- 房间能力：创建、加入、搜索、推荐、公开/私密房、基础房间资料。
- 角色体系：房主、管理员、上麦用户、听众。
- 礼物能力：普通礼物、连击礼物、节日礼物；SKU分低/中/高价格带。
- 榜单能力：贡献榜、消费榜的即时刷新。
- 风控能力：分钟级送礼限额、异常设备聚类、黑名单拦截。
- 账务能力：充值单、消费单、回滚单、T+1对账单。

**验收标准**
- 送礼请求成功率`>=99.5%`。
- 礼物动效端到端耗时`P95 < 800ms`。
- 订单幂等正确率`=100%`。
- 日对账差异率`<0.1%`。
- 异常订单可通过订单ID追踪最终态，审计日志保留`180天`。

### 6.3 REQ-003 群聊音频（8麦位，100并发）
**目标**
- 在房间多听少说的典型模式下，保证可听、可控、低延迟。
- 让房主和上麦用户在普通移动网络环境中可持续互动。

**核心流程**
1. 用户入房后建立WebRTC传输。
2. 上麦用户发布音频流，听众订阅音频流。
3. 服务端按活跃说话人与优先级下发订阅策略。
4. 网络波动时自动降级，恢复后自动补偿。

**功能需求**
- 麦位管理：上麦申请、下麦、抢麦冲突处理。
- 音频参数：Opus 24-32kbps，20ms ptime，启用DTX/FEC。
- 弱网处理：优先保活音频，低优先级订阅可降级到4路。
- 观测能力：延迟、抖动、丢包、卡顿时长、订阅降级事件。

**验收标准**
- `100并发（8麦 + 92听众）`下语音延迟`P95 < 300ms`。
- 上麦成功率`>=97%`。
- 5分钟内语音连续可听率`>=98%`。
- 弱网降级触发后`15秒`内恢复到可听状态的比例`>=95%`。

### 6.4 REQ-004 退出/掉线重连（30秒窗口）
**目标**
- 减少网络抖动、App前后台切换、短时断网带来的房间流失。
- 在恢复链路中保持关键状态一致，避免用户重复操作。

**核心流程**
1. 客户端识别信令断连或媒体断流，进入`RECONNECTING`。
2. 客户端携带`reconnect_token + last_seq`请求恢复。
3. 服务端校验恢复窗口和会话快照，决定恢复或要求重新入房。
4. 恢复成功后执行重订阅与关键事件回补。
5. 超过窗口则重新加入房间，并补偿关键状态。

**功能需求**
- 入房签发`reconnect_token`，默认TTL `30秒`。
- 记录会话快照：麦位、订阅列表、最近事件序号。
- 状态机：`CONNECTED -> RECONNECTING -> RECOVERED / REJOIN_REQUIRED`。
- 补偿机制：礼物、榜单、麦位等关键事件按序回放。
- 提示机制：用户可见的恢复提示与失败降级提示。

**验收标准**
- `30秒`恢复窗口内重连成功率`>=85%`。
- 平均恢复时长`<=8秒`，P95 `<=15秒`。
- 重连后关键状态一致性`>=99.9%`。
- 超出恢复窗口后仍能重新入房并获得关键状态补偿。
